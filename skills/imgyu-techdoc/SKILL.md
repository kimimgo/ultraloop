# 기술 문서 제작 하네스 (철학 + 방법)

> 목적: 주제·도메인 불문 — **단일 자립형 HTML**(외부 CSS 없음, 다이어그램 라이브러리만 CDN)
> 형태의 정돈된 기술 문서를 제로부터 작성한다. 개발 계획서·아키텍처/인프라 설계서·기술 제안서·
> RFC·설계 노트 등 어디에나 같은 **제작 철학과 방법**을 적용한다.
>
> 이 스킬은 **철학과 방법만** 담는다. 어떤 특정 문서의 내용·브랜드에 묶이지 않는다. 색 팔레트,
> 섹션 구성, 예시 문구는 전부 *교체 가능한 기본값*일 뿐이고, 불변인 것은 §0의 8가지 철학이다.

동반 레퍼런스 3개 — 필요한 단계에서만 읽어라:
- `reference/template.html` — 디자인 토큰 시스템 + 모든 컴포넌트/다이어그램 6종의 복붙용 스켈레톤
  (디자인 *방법*의 참조 구현). **이걸 베이스로 시작**하되 팔레트·브랜드는 프로젝트에 맞게 교체.
- `reference/diagrams.md` — Mermaid 쿡북(노드 모양·실선/점선 의미·subgraph 스윔레인·변경강조·이스케이프).
- `reference/style-and-tone.md` — 글쓰기 보이스(단정형·Phase 스코핑·"설계 의도" 평어 내러티브·
  단일기준·변경추적) + Do/Don't.

## 0. 문서 제작 철학 (이 8가지가 정체성 — 도메인이 바뀌어도 불변)

1. **단일 HTML · 외부 의존 0.** CSS는 `<style>` 인라인, 폰트는 시스템 스택, 유일한 외부 로드는
   다이어그램 라이브러리(Mermaid). 첨부 없이 파일 하나로 완결·이식.
2. **일관된 디자인 토큰(하우스 스타일).** `:root` 변수 팔레트 + 사이드바 TOC + 카드형 섹션 +
   비교표 + blockquote. 색/브랜드는 교체해도 *구조와 토큰 체계*는 유지한다.
3. **다이어그램은 의미를 담는다.** 노드 모양·실선/점선·스윔레인이 각각 약속된 뜻을 가진다.
   "그림"이 아니라 "읽는 도식"이다.
4. **단정형 평서 산문.** 결정된 건 "~한다/~다"로 못 박는다. 추측·공손체·빈 형용사 금지.
5. **⭐ "설계 의도" 평어 내러티브.** 적어도 한 곳에서 핵심 기술 선택을 *일상어 비유*로 풀고,
   **의도적으로 뺀 것(과설계 회피)**과 그 이유를 밝힌다. 이게 이 문서류의 시그니처다.
6. **Phase/범위 레이어링.** 모든 범위를 시간/조건으로 가른다: 지금(Day 1/P1) · 조건부(구조만
   확보) · 후순위(Phase 2~N) · 범위 외. 미확정은 **기한과 함께** 명시한다.
7. **단일 기준 규율.** 상세가 다른 문서/주체 소관이면 그것을 "단일 기준"으로 가리키고 여기선
   반복하지 않는다. 문서 비대화·중복 갱신을 막는다.
8. **정량 근거 + 변경추적(본문 오염 금지).** 숫자·날짜·구체 사양으로 못 박는다(빈 형용사 금지).
   **⭐ 본문은 항상 "현재 상태"만 단정형으로 기술한다 — 변경 이력을 본문에 섞지 않는다.**
   `"2026-07-03 변경"` 같은 날짜·개정 표식(`.chg-tag`)이나 `.changelog` 박스를 본문 섹션·표·산문
   안에 넣지 마라. 모든 변경 사항은 **문서 말미의 단일 "변경 이력" 별첨 섹션**에 `.changelog`로
   모은다(§3 패턴 라이브러리의 "변경 이력(별첨)"). 독자는 본문에서 "지금 시스템이 어떤가"만,
   별첨에서 "무엇이 언제·왜 바뀌었나"를 읽는다 — 두 관심사를 섞으면 본문이 개정 주석으로 오염된다.
   (다이어그램에서 강조가 꼭 필요하면 `classDef chg` 노드까지만 허용 — 산문·표엔 태그 금지.)

> 도메인이 데이터 파이프라인이든 모바일 앱이든 보안 아키텍처든, **이 8가지는 그대로** 적용된다.
> 바뀌는 건 섹션 구성·다이어그램 선택·색뿐이다(§3).

## 1. 절대 규칙 (기술적 함정)

1. 단일 HTML, 외부 CSS 0, 다이어그램 라이브러리만 CDN(SRI 무결성), 시스템 폰트 스택.
   (밀집 아키텍처도만 ELK 레이아웃 opt-in — 외부 의존 2개·SRI 불가, `reference/diagrams.md §ELK` 참조.)
2. 한국어 평서형(영문 문서면 영문 평서). 고유명사·경로·전문용어 verbatim — 번역·로마자화 금지.
3. **Mermaid 라벨 줄바꿈은 `&lt;br/&gt;`** (HTML 안 `<pre class="mermaid">`이라 `<`/`>`/`&` 이스케이프 필수).
4. **실선 `-->` = 필수/주 흐름, 점선 `-. 라벨 .->` = 조건부·페더레이션·비동기·전환.** 혼용 금지.
5. 결정은 단정, 미확정은 기한 명시. 헤지 금지.
6. CSS `<style>`·다이어그램 init 스크립트의 *구조*는 변형하지 않는다. 교체 허용 = `:root` 팔레트,
   그리고 init의 `theme:'base'` `themeVariables`(= `:root`와 1:1 매핑) 값·`themeCSS`. securityLevel·htmlLabels는 유지.
7. **다이어그램은 문서 팔레트와 일체화한다** — `theme:'base'`+`themeVariables`로 노드·엣지·폰트가 본문과
   같은 색/폰트 DNA를 공유한다. 종류별 의미는 모양 + 의미별 색 classDef(actor/external/store/chg, diagrams.md §7)로.
8. **⭐ 변경 이력은 본문 밖 별첨으로만.** 본문 산문·표에 `.chg-tag`(`"YYYY-MM-DD 변경"`)·`.changelog`
   박스를 넣지 마라. 변경은 문서 말미 단일 "변경 이력" 별첨 섹션에 `.changelog`로 모은다. 본문은
   현재 상태를 단정형으로 서술하고, "언제 바뀌었다"는 서술을 섞지 않는다. (다이어그램 `classDef chg`
   노드 강조는 예외로 허용 — 그래도 라벨에 날짜는 넣지 않는다.)

## 2. 고정 워크플로 (단계 스킵 금지)

### Step 1 — 입력 수집 & 스코프 확정
프로젝트명·한 줄 정의·헤더 메타 칩, 핵심 스택/구성, 기능·범위(지금/조건부/제외), 횡단 관심사
(인증·권한·상태·보안), 일정·우선순위, 위험·미확정+기한, 용어를 확보. 없으면 합리적 가정 후
"위험·미확정" 섹션에 **기한과 함께** 적는다.

### Step 2 — 섹션 아웃라인 승인 (게이트)
§3 패턴 라이브러리에서 **이 프로젝트에 필요한 섹션만 골라** 번호 매긴 목록을 제시하고 사용자
승인을 받는다. 어떤 다이어그램을 어디 넣을지도 합의. 작은 메모성 요청이면 생략 가능.

### Step 3 — 스켈레톤 복제
`reference/template.html`을 베이스로 복사. CSS·Mermaid init 구조는 그대로(팔레트만 교체 가능).
사이드바 TOC 링크와 `<section id="sN">`을 승인된 아웃라인에 맞춰 생성(1:1).
**이 스켈레톤(빈 섹션 골격)을 먼저 대상 파일로 저장(Write 1회).** 본문은 그 파일을 Edit로 채운다 —
처음부터 완성본 HTML을 한 번에 만들지 않는다.
⚠ 스켈레톤의 플레이스홀더(`{{…}}`)·기본 팔레트(`--brand`)·예시 라벨(SSO·앱 서비스·Primary/Standby 등)·
`lang="ko"`·한글 폰트 스택은 전부 *교체 가능한 기본값* — 대상 클라우드·브랜드·언어 용어로 빠짐없이 채운다.

### Step 4 — 섹션 본문 작성 (증분)
`reference/style-and-tone.md` 보이스로. 각 섹션은 대개 표(비교/정의) + 짧은 단정형 단락 +
필요 시 blockquote(설계 의도/정의).
**전체 HTML을 한 턴에 통째로 생성하지 말 것.** 저장된 스켈레톤을 열어 섹션을 Edit로 1~몇 개씩 채운다.
한 응답이 너무 길면 transient 오버로드(`API error · Retrying`)에 걸리고, 실패 시 문서 전체를 다시
만들어야 한다. 증분 작성은 각 요청을 작게 유지하고 재시도 비용을 한 섹션으로 한정한다.

### Step 5 — 다이어그램 삽입
`reference/diagrams.md`의 6종에서 프로젝트에 맞는 것만 선택. 변경 노드는 `classDef chg` + `class N chg;`.

### Step 6 — 변경 이력 별첨 (개정일 때만)
신규 문서면 생략. 개정이면 **문서 말미에 단일 "변경 이력" 별첨 섹션**을 만들어 그 안에서만
`.changelog` 박스로 날짜·항목·근거를 요약한다. **본문 섹션·표·산문에는 `.chg-tag`("YYYY-MM-DD 변경")도
`.changelog`도 넣지 않는다** — 본문은 현재 상태만 단정형으로 서술한다(§0-8·§1-8). 강조가 꼭 필요한
다이어그램 노드에 한해 `classDef chg`까지만 허용(라벨에 날짜 금지).

### Step 7 — 자체 검증(§4) 전부 통과 → 저장
파일명은 한글 허용(예: `<프로젝트>_<문서종류>.html`). 에이전트는 브라우저를 못 여니, 저장 전
**텍스트로** 모든 mermaid 블록을 점검한다: 괄호·따옴표 짝, `&lt;br/&gt;`·비교연산자(`&gt;`/`&lt;`)
이스케이프, 엣지가 참조하는 노드의 선언 여부. 그 뒤 사용자에게 브라우저 렌더 확인을 안내한다.

## 3. 섹션 패턴 라이브러리 (고정 목차가 아니라 메뉴)

필요한 패턴만 골라 조립한다. 각 패턴 = 권장 컴포넌트. 거의 모든 문서가 개요·시각 개념도·
우선순위/일정·용어로 시작·끝맺는다. 나머지는 문서 종류에 따라 가감.

| 패턴 | 언제 | 권장 컴포넌트 |
|------|------|----------------|
| 문서 개요 | 거의 항상 | 메타 표 + "비중 있게 다루는 N개 영역" 한 줄 |
| 시스템/서비스 개념도 | 거의 항상 | `flowchart TB` 스윔레인 + blockquote 한 문단 요약 |
| 전체 아키텍처 | 시스템 설계류 | `flowchart LR` 중첩 subgraph + 구성요소 역할표 + (해당 시) 운영/개발 분리표 |
| 인프라/배포 상세 | 인프라 비중 클 때 | 컴포넌트 소절 + 비교표 + **설계 의도 blockquote** + 네트워크 `flowchart TB` (변경 이력은 본문에 넣지 말고 별첨 섹션으로) |
| 개발 방식 & CI/CD | 구현/배포 다룰 때 | 모듈 분리 원칙 불릿 + 브랜치 표 + CI/CD `flowchart LR`(게이트 다이아몬드) |
| 도메인/데이터 설계 | 앱·데이터 설계류 | 설계 원칙 단락(상세는 단일 기준 위임) + 도메인 표 |
| 기능 범위 | 제품/기능 정의류 | 기능표 + 조건부/범위외 + "상세는 X 단일 기준" |
| 인증·신원 / 권한(RBAC) | 보안·접근통제 | 항목/내용 표 + 권한 종류 표 + 흐름 표 |
| 상태/워크플로 정의 | 상태기계 있을 때 | `stateDiagram-v2` + 상태/의미 표 |
| 데이터 플랫폼/분석 | 데이터 파이프라인 | 목적/처리/산출물 표 + OLTP vs OLAP 분리 |
| 모델/평가지표 *(ML 도메인 예)* | ML 포함 | 지표 정의표(AUC-PR·recall@FPR 등) + 임계값 표 + 평가 데이터 단일기준 |
| MLOps/재학습 *(ML 도메인 예)* | 모델 운영 | drift 트리거 표 + 재학습 `gantt` + 게이트 다이아몬드 |
| 조건부 기능 | 외부 의존 불확실 | 의존성·기한 + 소절별 "구조만 확보" |
| 우선순위 & 일정 | 거의 항상 | P1/P2/P3 `.badge` 표 + `gantt` + 순연 정책 |
| 보안 | 보안 관심사 | 영역/조치 표 |
| 위험·미확정 | 거의 항상 | 항목/내용/대응/**기한** 표 |
| 명시적 제외(Out of Scope) | 범위 경계 필요 | Phase별 이관 불릿 |
| **변경 이력(별첨)** | **개정 문서일 때** | **문서 말미 단독 섹션 + `.changelog` 박스(날짜·항목·근거). 본문의 모든 변경 서술을 여기로 모은다 — 본문엔 `.chg-tag`/`.changelog` 금지** |
| 용어(글로서리) | 거의 항상 | 용어/정의 표 |

## 4. 자체 검증 체크리스트 (저장 전 전부 확인)

- [ ] 단일 HTML, 외부 CSS 0, 다이어그램 라이브러리만 로드, 시스템 폰트 스택.
- [ ] CSS·다이어그램 init 구조 미변형(`:root`·`themeVariables`·`themeCSS` 값 교체는 허용).
- [ ] 다이어그램이 `theme:'base'`+`themeVariables`로 문서 팔레트·폰트와 일체화(회색 neutral 아님).
- [ ] 노드 종류가 의미별 색 classDef(actor/external/store/chg) + 모양으로 구분, 문서에 색 범례 동반.
- [ ] 각 다이어그램에 `accTitle`/`accDescr`(스크린리더 대응).
- [ ] 사이드바 TOC 링크 ↔ `section id` 1:1, 번호 연속.
- [ ] 모든 Mermaid 라벨 줄바꿈이 `&lt;br/&gt;`, 미이스케이프 `<`/`>`/따옴표 없음.
- [ ] 실선(필수)·점선(조건부/페더레이션/비동기) 구분 일관.
- [ ] 변경 노드 `classDef chg`+`class … chg;`(개정 시).
- [ ] 결정 단정형, 미확정 기한 명시, 헤지 없음.
- [ ] 비교표·위험표(기한 열)·용어 글로서리 존재(해당 섹션 쓸 때).
- [ ] 우선순위/변경 강조가 색 외 텍스트 라벨(P1/P2/P3, "변경" 배지)도 동반 — 흑백 인쇄·색맹 대응.
- [ ] **최소 1개 "설계 의도" blockquote가 평어로 WHY + 의도적 제외(과설계 회피)를 설명.**

- [ ] **본문(모든 `<section>`) 산문·표에 `.chg-tag`("YYYY-MM-DD 변경")·`.changelog` 박스가 없다.**

- [ ] §0의 8가지 철학이 도메인과 무관하게 반영됨.

## 5. 출력 규칙

- 사용자에게는 **무엇을 만들었는지 + 파일 경로 + 열어보는 법**만 간결히. 긴 HTML 채팅 덤프 금지.
- **증분 빌드 강제(중요):** 스켈레톤 1회 Write → 섹션별 Edit. 완성본 HTML을 한 응답에 통째로 쏟지
  않는다. 큰 단일 출력은 API 오버로드·재시도·진행분 손실의 주원인이다. 문서가 길수록 더 잘게 나눈다.
- 아웃라인 승인 게이트(Step 2)는 건너뛰지 마라(작은 메모 제외).
- 다이어그램 6종을 다 쓸 필요 없다 — 프로젝트에 맞는 것만. 단 시스템 개념도와 일정 간트는 사실상 필수.
- 도메인이 낯설면 §0 철학은 유지하되, 섹션·다이어그램·색·용어는 그 도메인 관행에 맞춰 조정한다.
